Benefits and Challenges

Understanding the trade-offs of micro-frontend architecture

Benefits

  • ✓ Independent deployment and scaling
  • ✓ Team autonomy and technology diversity
  • ✓ Fault isolation and resilience
  • ✓ Gradual migration from monoliths
  • ✓ Code reusability across teams

Challenges

  • ✗ Increased complexity and coordination
  • ✗ Bundle size and performance overhead
  • ✗ Cross-team communication and contracts
  • ✗ Testing and debugging complexity
  • ✗ Shared dependencies and versioning

Communication Patterns

How micro-frontends communicate with each other

Event Bus

// event-bus.js
class EventBus {
  constructor() {
    this.events = {};
  }
  
  on(event, callback) {
    if (!this.events[event]) {
      this.events[event] = [];
    }
    this.events[event].push(callback);
  }
  
  emit(event, data) {
    if (this.events[event]) {
      this.events[event].forEach(callback => callback(data));
    }
  }
}

// Global event bus
window.eventBus = new EventBus();

// Usage in micro-frontend
window.eventBus.emit('user-login', { userId: 123 });
window.eventBus.on('user-login', (data) => {
  console.log('User logged in:', data);
});

Shared State

// shared-state.js
class SharedState {
  constructor() {
    this.state = {};
    this.subscribers = [];
  }
  
  setState(newState) {
    this.state = { ...this.state, ...newState };
    this.notifySubscribers();
  }
  
  getState() {
    return this.state;
  }
  
  subscribe(callback) {
    this.subscribers.push(callback);
    return () => {
      this.subscribers = this.subscribers.filter(sub => sub !== callback);
    };
  }
  
  notifySubscribers() {
    this.subscribers.forEach(callback => callback(this.state));
  }
}

// Global shared state
window.sharedState = new SharedState();

PostMessage API

// Parent window
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://trusted-domain.com') return;
  
  switch (event.data.type) {
    case 'USER_LOGIN':
      handleUserLogin(event.data.payload);
      break;
    case 'NAVIGATE':
      navigateToRoute(event.data.payload.route);
      break;
  }
});

// Child window (iframe)
parent.postMessage({
  type: 'USER_LOGIN',
  payload: { userId: 123, token: 'abc123' }
}, 'https://parent-domain.com');

Custom Events

// Custom event for micro-frontend communication
const userLoginEvent = new CustomEvent('microfrontend:user-login', {
  detail: {
    userId: 123,
    userData: { name: 'John Doe', email: 'john@example.com' }
  }
});

// Dispatch event
window.dispatchEvent(userLoginEvent);

// Listen for event
window.addEventListener('microfrontend:user-login', (event) => {
  console.log('User logged in:', event.detail);
  updateUserInterface(event.detail);
});

Deployment Strategies

Different approaches to deploying micro-frontends

Independent Deployment

Each micro-frontend is deployed independently with its own CI/CD pipeline.

# Each micro-frontend has its own deployment
# auth-service
docker build -t auth-service .
docker push auth-service:latest

# dashboard-service  
docker build -t dashboard-service .
docker push dashboard-service:latest

# shell-app
docker build -t shell-app .
docker push shell-app:latest

Blue-Green Deployment

Deploy new versions alongside existing ones and switch traffic instantly.

# Blue-Green deployment strategy
# 1. Deploy new version to green environment
kubectl apply -f auth-service-green.yaml

# 2. Run health checks
kubectl get pods -l app=auth-service-green

# 3. Switch traffic from blue to green
kubectl patch service auth-service -p '{"spec":{"selector":{"version":"green"}}}'

# 4. Clean up blue environment
kubectl delete deployment auth-service-blue

Canary Deployment

Gradually roll out new versions to a subset of users.

# Canary deployment with Istio
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: auth-service
spec:
  http:
  - match:
    - headers:
        canary:
          exact: "true"
    route:
    - destination:
        host: auth-service
        subset: v2
  - route:
    - destination:
        host: auth-service
        subset: v1
      weight: 90
    - destination:
        host: auth-service
        subset: v2
      weight: 10

Best Practices

Guidelines for successful micro-frontend implementation

Design Principles

  • ✓ Keep micro-frontends small and focused
  • ✓ Define clear boundaries and contracts
  • ✓ Use consistent design systems
  • ✓ Implement proper error boundaries
  • ✓ Plan for shared dependencies

Technical Guidelines

  • ✓ Use semantic versioning for APIs
  • ✓ Implement proper testing strategies
  • ✓ Monitor performance and errors
  • ✓ Document communication protocols
  • ✓ Plan for graceful degradation

Team Organization

  • ✓ Assign clear ownership of micro-frontends
  • ✓ Establish communication channels
  • ✓ Create shared component libraries
  • ✓ Implement code review processes
  • ✓ Plan for knowledge sharing
